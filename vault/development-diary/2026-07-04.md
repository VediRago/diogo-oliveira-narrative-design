# 2026-07-04 — Portfolio Architecture Locked

## Main Purpose

Today marked the point where the portfolio stopped being an architecture experiment and became an implementation project.

The goal was to lock the layer structure clearly enough that future work can focus on filling repositories instead of redesigning the system around them.

## Layer Structure Locked

The current portfolio structure is now:

```text
Layer 1
Home

Layer 2
UX Portfolio
Narrative Design
Noeme Systems

Layer 3
NOEME
The Founding Era
Joe

Layer 4
CEM
```

The important change is that every repository now has a clearly defined question it exists to answer.

- Layer 1 answers: Who is this, and where do I go from here?
- Layer 2 answers: How is this work approached?
- Layer 3 answers: What does that approach produce?
- Layer 4 answers its own singular question when a project cannot be described cleanly through the shared Layer 2 or Layer 3 structures.

This means the architecture can now act as a constraint instead of remaining an open design problem.

## UX Became The Reference Implementation

The UX Portfolio is now the reference implementation for Layer 2.

Rather than giving UX, Narrative Design, and Noeme Systems separate structures, all three now share the same Layer 2 architecture:

- README
- Case Studies
- Supporting Files
- Vault

The repositories differ by discipline, not by structural logic.

This removes repeated structural decisions from future work and makes implementation easier.

## Case Study Blueprint Updated

The Case Study Blueprint reached a stable second version.

The current structure is:

```text
Method

↓

Explanation

↓

Application

↓

Evidence

↓

Continue
```

The main correction was that Application does not have to be prose.

Sometimes the clearest application is a dependency chain, layered system, decision tree, progressive refinement, or cause/effect structure.

The important distinction is:

```text
Chronology ≠ Causality.
```

Chronology should usually be removed unless the order itself is meaningful.

Dependency should be preserved when it proves the method.

## Narrative Design Refit

With the Layer 2 Blueprint established, Narrative Design began being refitted to match the same architecture.

The current reading path is:

```text
Building a World to Test Quest Design

↓

Building a Modular Narrative Method

↓

Testing the Method Through Layered Reactivity

↓

Building NPCs Through Layered Pressure
```

The previous Noeme layered narrative case study no longer belongs inside Narrative Design.

It will move into Noeme Systems, where it fits better as production methodology.

Narrative Design should keep transferable methods.

Noeme Systems should keep the production systems behind NOEME.

## Method Separated From Product

The Founding Era was also separated from Narrative Design conceptually.

Narrative Design should not store the story itself.

The Founding Era belongs in Layer 3 because it presents product, not method.

This keeps the repository focused:

```text
Narrative Design = how the work is approached.
The Founding Era = what the work produced.
```

That separation makes both repositories easier to understand.

## Current State

The architecture is now stable.

Future work should focus on implementation rather than redesign.

The next major task is:

```text
Building NPCs Through Layered Pressure
```

After that, the Noeme layered narrative system should be moved into Noeme Systems and adapted to the same Case Study Blueprint v2.

From this point forward, the portfolio should evolve by filling the existing structure rather than changing the structure itself.
